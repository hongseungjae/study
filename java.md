## 객체지향이 뭐냐?
객체지향은 현실세계의 어떤 사물들을 보고 모델링하는 기법 입니다.
데이터와 함수를 묶어서 클래스라는 곳에 정의하여 설계할수있으며, 필드를 직접적으로 드러나게 하지 않고, 함수에 의해 접근할 수있게하여, 캡슐화가 잘 지켜질수있습니다.

이에 반해 절차지향언어는 데이터와 함수를 분리시키기때문에 이곳저곳에서 함수를 호출하여 데이터를 변경시킬수있습니다.
유지보수의 어려움이 있습니다

F : 구체적으로 EJB 기능 -> 어떤 기능을 쓰려면 상속을 받아야함. 상속받으면 매우 강한 결합이 생김. 상속을 강제하니까 지침 애플리케이션이 매우 무거움 더이상 할수있는게 제한적임 -> POJO 상속을 안받고도 프레임워크의 이점을 누리는것을 어필

따라서, Java 클래스 선언으로 특별히 상속받지 않는것 -> POJO Plain Old Java Object

## SOLID란 무엇이고, 이에 대해서 아는대로 설명하시오.
* 내 대답 : 객체지향 설계를 하는데 좀 더 객체지향스럽게 설계하는 방법이고, SRP, OCP, LIP, ISP, DIP 
* 일단 SRP는 싱글 리스퍼빌리티 , 책임이 하나여야 된다는것이고, 한 클래스가 하나의 책임을 가지고 있고, 그다음에 그것이 변경이 일어난다면 이 하나의 책임으로 인해 변경이 일어나야된다는것입니다. 그니깐 책임을 여러개가지고 있으면 전혀 상관없는 것이 영향을 받을수있기때문 입니다.
  - 이 말의 뜻은 책임이란 해당 클래스가 변경될 때 하나의 이유여야만 한다는 것입니다.
  - 어떤 서비스계층의 클래스가 db접근하는 코드와 db에 쿼리를 날리는 코드가 있다고 가정해보자

* OCP는 기능을 추가할때는 클라이언트의 코드에서 변경이 없이 추가할 수 있고, 영향을 안끼치고, 또한 주입으로 자유롭게 바꿔낄수 있는것입니다.
  - OCP 원칙의 가장 잘 따르는 예시가 바로 자바의 데이터베이스 인터페이스인 JDBC입니다.
  - JDBC인터페이스를 통하여 데이터베이스를 
  - 객체를 직접 수정하지 않고도 변경사항을 적용할 수 있도록 설계해야 한다.


* LSP 리스코프 치환원칙은 상속 관계일때 부모 객체를 받든, 자식 객체를 받든, 온전히 수행할수있어야 되는것을 말합니다. 그게 안될경우 instanceof로 구별을해주어야 하는 그런 문제들이 발생합니다.


* ISP 인터페이스 분리 원칙으로, 인터페이스에 여러 책임을 가지고 있으면 안되는것이고 각각하나의 책임만 가지고 분리하는 것입니다.
  - 핵심 : 고수준 모듈이 저수준 모듈에 의존하지 말아야 한다.


* DIP는 의존성 역전 원칙은, 구체적인것말고 추상적인것에 의존하는 것을 말합니다. 구체적인것에 의존하면 깨지기가 쉽기 때문이고, 꺠지기 쉽다는것은 어떤 변경이 있을때 코드를 직접수정하여야 합니다.

## TDD란 무엇이고, 이에 대해서 아는대로 설명하시오.
